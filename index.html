Разработка ПО

Разработка ПО

Конструирование — главный этап разработки ПО, без которого не обходится ни один проект.

Основные этапы конструирования:

  • детальное проектирование
  • кодирование
  • отладка
  • интеграция и тестирование приложения разработчиками (блочное тестирование и интеграционное тестирование).
Метафоры

Метафоры

Метафоры помогают понять процесс разработки ПО, сопоставляя его с другими, более знакомыми процессами.

Виды:

  • Литературная метафора: написание кода
  • Сельскохозяйственная метафора: приращение системы
  • Метафора жемчужины: медленное приращение системы (аккреция ПО)
  • Строительная метафора: построение ПО
Подготовка

Подходы к разработке ПО

Последовательный

  • Требования довольно стабильны
  • Проект приложения прост и относительно понятен
  • Разработчики знакомы с прикладной областью
  • Проект не связан с особым риском
  • Важна долговременная предсказуемость проекта
  • Затраты на изменения требований окажутся скорее всего высокими

Итеративный

  • Требования относительные непонятны или нестабильны
  • Проект приложения сложен и не ясен
  • Разработчики не знакомы с прикладной областью
  • Проект сопряжен с высоким риском
  • Долговременная предсказуемость проекта не играет особой роли
  • Затраты на изменения требований, скорее всего, будут низкими

Управление сложностью

Главный технический императив ПО

Не следует пытаться охватить всю программу сразу. Вместо этого мы должны попытаться организовывать программы так, что бы можно было безопасно работать с их отдельными фрагментами по очереди.

Дийкстра, 1972

Причины усложнения:
  • Сложное решение простой проблемы
  • Простое, но неверное решение сложной проблемы
  • Неадекватное сложное решение сложной проблемы
Подходы к управлению сложностью:
  • Старайтесь свести к минимуму объем существенной сложности, с которым придется работать в каждый конкретный момент времени
  • Сдерживайте необязательный рост несущественный сложности

Желательные характеристики проекта

  • Минимальная сложность
  • Простота сопровождения
  • Слабое сопряжение
  • Расширяемость
  • Возможность повторного использования
  • Портируемость
  • Высокий коэффициент объединения по входу
  • Низкий или средний коэффициент разветвления по выходу
  • Минимальная, но полная функциональность
  • Стратификация
  • Соответствие практикам, соглашениями и методикам

Семантическое сопряжение

Классы

АТД

Абстрактный тип данных - это набор, включающий данные и выполняемые над ними операции.
Преимущества использования АТД
  • Возможность сокрытия деталей реализации
  • Ограничение области изменений
  • Более высокая информативность интерфейса
  • Легкость оптимизации кода
  • Легкость проверки кода
  • Удобочитаемость и понятность кода
  • Ограничение области использования данных
  • Возможность работы с сущностями реального мира, а не с низкоуровневыми деталями реализации

Хорошая абстракция

  • Выражайте в интерфейсе класса согласованный уровень абстракции
  • Убедитесь что понимаете, реализацией какой абстракции является класс
  • Предоставляйте методы вместе с противоположными им методами
  • Убирайте постороннюю информацию в другие классы
  • Делайте интерфейсы программными, а не семантическими
  • Опасайтесь нарушения целостности интерфейса при изменении класса
  • Не добавляйте публичные свойства которые не согласуются с абстракцией интерфейса
  • Рассматривайте абстракцию и связность вместе

Хорошая инкапсуляция

  • Минимизируйте публичные методы и свойства
  • Не делайте предположений о клиентах класса
  • Цените легкость чтения кода выше, чем удобство написания
  • Очень, очень настороженно относитесь к семантическим нарушениям инкапсуляции
  • Остерегайтесь слишком высокого сопряжения (high coupling)

  • Ограничение влияния изменений

  • Создание центральных точек управления
  • Облегчение повторного использования кода
  • Планирование создания семейств
  • Упаковка родственных операций / команд
  • Выполнение специфического вида рефакторинга

Удачные имена методов

  • Название должно описывать то что выполняет метод
  • Не использовать неоднозначные, расплывчатые, не выразительные глаголы в названии
  • Не использовать номера в названии методов для дифференциации / разделению на шаги
  • Не ограничивать длину метода искусственно
  • Для названия метода можно использовать описание возвращаемого значения
  • Метод можно именовать по схеме глагол + имя данных с которыми работает метод
  • Дисциплинированно использовать антонимы
  • Использовать конвенции именования

Советы по использованию аргументов

  • Если несколько методов используют одинаковые аргументы, то необходимо передавать их в одинаковом / согласованном порядке
  • Не передавать не используемые аргументы
  • Переменные статуса или ошибки передавать последними, так как они второстепенны по отношению к главной цели метода
  • Не изменять входные параметры и не переприсваивать
  • Ограничивать число аргументов не более 7, иначе следует передавать их как объект
Защитное программирование

Утверждения (assertion)

Утверждение - код, который проверяет правильность выполнения программы.
Утверждения не предназначены для показа ошибок в продакшн режиме, они для дев режима и разработки.
Общие принципы использования утверждений
  • Используйте процедуры обработки ошибок для ожидаемых событий и утверждения для событий, которые происходить не должны
  • Старайтесь не помещать выполняемый код в утверждения
  • Используйте утверждения для документирования пред- и постусловий
  • Для большей устойчивости проверяйте утверждения, а затем все равно обрабатывайте ошибки

Способы обработки ошибок

Выбор того или иного способа обработки ошибок зависит от того что важнее, Корректность или Устойчивость
  • Вернуть нейтральное значение
  • Заменить следующий корректным блоком данных
  • Вернуть последний успешный результат
  • Подставить ближайшее допустимое значение
  • Записать лог ошибки
  • Вернуть код ошибки
  • Вызвать обработчик ошибок
  • Показать сообщение об ошибке
  • Обработать ошибку в месте возникновения
  • Прекратить выполнение программы

Исключения

  • Используйте исключения для оповещения других частей программы об ошибках
  • Генерируйте исключения для действительно исключительных ситуаций и не используйте исключения по мелочам
  • Избегайте выкидывание исключение в конструкторах
  • Исключение должно содержать все необходимую информацию о причинах исключения
  • Избегать пустых блоков catch
  • Стандартизируйте работу с исключениями на проекте

Изоляция повреждений / баррикада

Публичные методы предполагают что получаемые данные небезопасны и отвечают за их проверку и исправление. Здесь используются обработчики ошибок.
Приватные методы предполагают что полученные данные прошли проверку в публичном методе и безопасны. Здесь используются утверждения

Конвенции именования

Зачем нужны конвенции

  • Они позволяют больше принимать как данное
  • Они помогают использовать знания полученные при работе над прошлыми проектами
  • Они ускоряют изучение кода нового проекта
  • Подавляют размножение похожих имен (totalRecords и recordsTotal)
  • Они могут компенсировать слабости языка (можно имитировать функционал который язык не поддерживает)
  • Они поддерживают связи между данными (recordId и recordName)

Тестирование, выполняемое разработчиками

Тестирование

  • Цель тестирования противоположна целям других этапов разработки.
  • Тестирование никогда не доказывает отсутствие ошибок
  • Тестирование не повышает качество ПО
  • Тестирование требует что бы вы рассчитывали найти ошибки в своем коде.

Приемы тестирования

  • Неполное тестирование
  • Структурированное базисное тестирование
  • Угадывание ошибок
  • Анализ граничных значений
  • Классы плохих данных
  • Классы хороших данных
Отладка

Поиск дефектов

  • Стабилизация ошибки
  • Определение источника ошибки
    • Сбор данных, приводящих к дефекту
    • Анализ собранных данных и формулирование гипотезы, объясняющей баг
    • Определение способа подтверждения или опровержения гипотезы, основанного на тестировании или изучении программы
  • Фикс бага
  • Тестирование фикса
  • Поиск похожих багов

Советы по поиску причин багов

  • Формулируя гипотезу используйте все имеющиеся данные
  • Воспроизведите ошибку несколькими способами
  • Используйте результаты отрицательных тестов для отсеивания гипотез
  • Используйте мозговой штурм
  • Сократите подозрительную область, где может быть баг
  • Поищите баг в функционале в котором уже встречались баги
  • Проверьте код который был изменен недавно
  • Расширьте подозрительный фрагмент кода
  • Обсудите баг с кем то
  • Установите лимит времени для быстрой и грязной отладки.
Рефакторинг

Разумные причины выполнения рефакторинга

  • Код повторяется
  • Метод слишком велик
  • Цикл слишком велик или имеет глубокую вложенность
  • Класс имеет плохую связность
  • Интерфейс класса не формирует согласованную абстракцию
  • Метод принимает слишком много параметров

Плохие причины рефакторинга

  • Не рассматривайте рефакторинг как оправдание написание плохого кода с намерением исправить его позднее
  • Не рассматривайте рефакторинг как способ избежать переписывание кода

Безопасный рефакторинг

  • Сохраняйте первоначальный код
  • Ограничивайте объем отдельных видов рефакторинга
  • Выполняйте отдельные виды рефакторинга по одному за раз
  • Составьте список действий которые хотите предпринять
  • Составьте список видов рефакторинга которые следует выполнить позже
  • Часто создавайте контрольные точки
  • Выполняйте регрессивное-тестирование
  • Создавайте дополнительные тесты
  • Выполняйте обзоры измненеий
  • Изменяйте подход в зависимости от рискованности рефакторинга

Стратегии рефакторинга

  • Выполняйте рефакторинг при создании новых методов
  • Выполняйте рефакторинг при создании новых классов
  • Выполняйте рефакторинг при исправлении дефектов
  • Выполняйте рефакторинг модулей подтверженных ошибках
  • Выполняйте рефакторинг сложных модулей

Мифы оптимизации

  • Сокращение числа строк высокоуровневого кода повышает быстродействие
  • Одни операции, вероятно, выполняются быстрее других
  • Оптимизацию следует выполнять по мере написания кода
  • Быстродействие программы не менее важно чем ее корректность

Алгоритм оптимизации

  • Напишите хороший и понятный код, поддающийся легкому изменению
  • Если производительность не устраивает
    • Закоммитьте рабочий код
    • Оцените производительность системы и найдите горячие точки
    • Определите чем обусловлена низкая производительность
    • Оптимизируйте проблемное место
    • Оцените каждую оптимизацию по одной за раз
    • Если оптимизация не привела к улучшению, повторите все заново
